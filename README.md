# Call Me Maybe

## Description
This project explores constrained decoding techniques applied to a small (0.6B parameter model) LLM in order to reliably generate structured JSON outputs. The main goal is to ensure that the model produces valid and well-formed JSON objects while still leveraging the generative capabilities of the LLM.

Instead of post-processing or correcting invalid outputs, this approach constrains the decoding process itself. By dynamically filtering which tokens the model is allowed to generate at each step, the system guarantees structural correctness during generation.

The project combines:
- Token-level filtering using predicates
- Logit masking to enforce valid token selection
- A state machine to track JSON structure in real time

## Algorithm Explanation

The core of this project is a constrained decoding pipeline that ensures only valid tokens are generated at each step.

### 1. Token Filtering via Predicate

A method is used to retrieve valid tokens based on any condition.
This allows dynamic filtering depending on the current generation state.

### 2. Logits Masking

Invalid tokens are removed by masking their logits.
This ensures the model can only select from valid tokens.

### 3. State Machine for JSON Generation

A state machine is used to track:
- Whether we are inside a string
- Whether a key or value is expected
- Current JSON nesting level
- Current key being processed

Based on this state, valid tokens are computed dynamically.

The generation loop:
- Requests valid tokens from the predicate system
- Masks logits accordingly
- Updates the state after each token
- Stops when a complete JSON object is formed

### 4. Hybrid Function Selection

Before JSON generation, a function is selected using the LLM:

This ensures:
- Correct parameter names
- Proper type enforcement (string, number)

The final JSON is then generated using constrained decoding, guided by the selected function definition.

## Design Decisions

The main design decision was to fully embrace constrained decoding rather than relying on post-processing or partial templating.

Key choices:
- Generating the entire JSON with the LLM instead of hardcoding structure
- Using a predicate-based token filtering system for flexibility
- Implementing a state machine to enforce structure correctness
- Keeping a hybrid approach for function selection to guarantee parameter consistency

This approach allowed deeper understanding of how token-level control affects generation.

## Performance Analysis

- Accuracy: High in most cases, with correct JSON structure and parameter typing
- Speed: Fast despite multiple checks per token
- Reliability: Very stable; the program rarely crashes and consistently produces usable outputs

Some edge cases exist where unusual characters in prompts can break JSON formatting, but these are rare.

## Challenges Faced

- Ensuring that the "parameters" key always maps to a correctly structured JSON object
- Tracking the current key to enforce correct parameter names and types
- Designing robust conditions for token validation
- Transitioning from an initial approach (LLM filling predefined JSON) to full constrained decoding

The solution involved:
- Implementing a detailed state tracker
- Enforcing structure through token constraints rather than corrections
- Keeping a hybrid function-selection step for reliability

## Testing Strategy

Testing was performed using:
- Edge cases generated by a larger LLM
- Various prompt formats to stress-test JSON validity
- Manual validation of output structure and correctness

This helped identify weaknesses in token conditions and improve robustness.
